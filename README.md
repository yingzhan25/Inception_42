# Inception

## Description

This project, "Inception," involves setting up a small infrastructure of various services using Docker and `docker-compose`. The primary goal is to gain a deeper understanding of containerization, system administration, and network management by deploying a multi-container application.

The core setup includes a LEMP stack (Linux, Nginx, MariaDB, PHP) running a WordPress website. Additionally, several bonus services are included for monitoring, caching, and administration.

### Docker and Project Structure

-   **Docker**: Each service (e.g., Nginx, MariaDB, WordPress) is encapsulated in its own Docker container. This approach ensures that each service runs in an isolated environment with its own dependencies, making the application portable, scalable, and easy to manage.
-   **Docker Compose**: The `srcs/docker-compose.yml` file is the orchestrator for this project. It defines all the services, networks, and volumes, and manages the lifecycle of the entire application stack.
-   **Source Files**: The `srcs/requirements` directory contains the `Dockerfile` and necessary configuration files for building each service's container image.

### Design Choices & Comparisons

#### Virtual Machines vs. Docker

-   **Virtual Machines (VMs)** virtualize the hardware, meaning each VM runs a full copy of an operating system on top of a hypervisor. This leads to significant overhead in terms of disk space, CPU, and RAM.
-   **Docker** virtualizes the operating system. Containers are lightweight, isolated processes that share the host machine's kernel but have their own filesystem and network space. This results in much faster startup times, lower resource consumption, and greater density (more containers can run on a host than VMs). For this project, Docker is ideal as it allows for a complex stack of services to run efficiently even on a single machine.

#### Secrets vs. Environment Variables

-   **Environment Variables (`.env` file)** are used in this project to manage configuration details like user names. They are easy to use and are injected into the container's environment at runtime. However, they can be a security risk as they can be inspected by anyone with access to the container or the host's Docker daemon.
-   **Docker Secrets** are a more secure way to manage sensitive data. They are used to store password in this project. Secrets are encrypted and only mounted into the container's memory (as files in `/run/secrets/`) when a service is granted access. They are not stored in the container's environment or on disk unencrypted. While more secure, they are primarily designed for Docker Swarm mode. For the scope of this project, environment variables were chosen for their simplicity with `docker-compose`.

#### Docker Network vs. Host Network

-   **Host Network**: Using the host network (`network_mode: host`) would make the container share the host's networking namespace. A port exposed in the container would be exposed on the host's IP address. This can simplify some network configurations but sacrifices the isolation that is a key benefit of Docker.
-   **Docker Bridge Network**: This project uses a custom bridge network (`inception_network`). This creates an isolated network for all the services. Containers on the same bridge network can communicate with each other by their service name (e.g., `wordpress` can reach `mariadb` at `http://mariadb:3306`). This is more secure and organized, as services are not exposed to the outside world unless explicitly published via the `ports` directive.

#### Docker Volumes vs. Bind Mounts

-   **Bind Mounts** map a file or directory on the host machine to a directory in the container. The path is managed by the user. While useful for development (e.g., mounting source code), the project requirements for persistent data specifically forbid their use for the database and WordPress files.
-   **Docker Volumes** are the preferred mechanism for persisting data generated by and used by Docker containers. In this project, we use **Named Volumes** (`mariadb_data`, `wordpress_data`). These volumes are managed by Docker itself. To meet the requirement of storing data in a specific host path (`/home/yingzhan/data/...`), the volumes are configured with the `local` driver and `bind` options. This technique creates a Docker-managed named volume that is physically located at a specific host path, satisfying all project constraints.

## Instructions

1.  **Prerequisites**:
    *   Docker and Docker Compose must be installed.
    *   You must have `sudo` rights.

2.  **Setup**:
    *   The `Makefile` requires the data directories to be located at `/home/yingzhan/data/`. If your username is different, you must update the paths in the `Makefile` and `srcs/docker-compose.yml`.
    *   Create a `.env` file in the `srcs` directory with the necessary environment variables (e.g., `DB_NAME`, `DB_USER`, etc.).
    *   Create a `my_password.txt` file under the `secrets` subdirectory in the `srcs` directory with the password.

3.  **Execution**:
    Use the provided `Makefile` at the root of the repository:
    *   `make all` or `make up`: Builds the container images and starts all services in detached mode.
    *   `make down`: Stops and removes all containers, networks, and volumes defined in the compose file.
    *   `make clean`: Stops and removes everything, and also prunes all unused Docker images.
    *   `make fclean`: Performs a `clean` and also deletes the contents of the WordPress and MariaDB data directories on the host machine.
    *   `make re`: A shortcut for `fclean` followed by `all`. Rebuilds and restarts everything from scratch.

4.  **Accessing Services**:
    *   **WordPress**: `https://localhost`
    *   **Adminer**: `http://localhost:8080` (Database management tool)
    *   **cAdvisor**: `http://localhost:8082` (Container monitoring)
    *   **Static Page**: `http://localhost:8081` (Served by Lighttpd)
